**File:** `store/useAppStore.ts`

*   **Type Definitions:**

    *   `AppStore`:
        *   `pendingAuthAction`: `PendingAuthAction`

        *   `clearPendingAuthAction`: `() => void`
*   **Store Factory:** Use `import { create } from 'zustand'` to construct `useAppStore`. Initialize `pendingAuthAction` as `null`.

*   **Logic:** Upon a successful sign-in redirect execution, check the Zustand store. If `pendingAuthAction === "save-meal"`, immediately re-trigger or allow the deferred action, then invoke `clearPendingAuthAction()`.

## 3. Meal Detail Screen Component
**File:** `(tabs)/index.tsx`

Refactor the saving interface to account for loading boundaries, local mutation states, and global auth safeguards.

### A. State & Hooks Setup
*   **Zustand Hooks:** Extract `setPendingAuthAction` from `useAppStore`.
*   **Convex Hooks:** 
    *   Bring in your `saveMeal` mutation hook and `getSavedMeals` query hook.
    *   Use `useConvexAuth()` to extract `isLoading` (renamed contextually to `isConvexAuthLoading`) and `isAuthenticated`.
*   **Memoized Identifiers:** Map out a `savedMealsIds` array (or `Set` for $O(1)$ lookups) from your query dataset using `useMemo`.

### B. Button Local UI State
*   Maintain a local component state `isMutationLoading: boolean` (initialized to `false`) to track the exact lifecycle duration of the `saveMeal` network call.

### C. Button Disabled & Text States
Compute the button configuration deterministically based on the following precedence rules:

| Condition | Text Label | Disabled State |
| :--- | :--- | :--- |
| `isSaved` (ID exists in memoized set) | "Saved" | `true` |
| `isMutationLoading` (Active database write) | "Saving..." | `true` |
| `isConvexAuthLoading` | "Connecting..." | `true` |
| `!isAuthenticated` | "Save Meal" | `false` (Triggers redirect logic) |
| *Default State* | "Save Meal" | `false` |

### D. On-Press Event Handler (`handleSaveMeal`)
Implement the execution sequence with the following logical short-circuits:

1.  **Duplicate Guard:** If `isSaved` is true, immediately return/abort.
2.  **Authentication Handshake:**
    *   If `!isAuthenticated`:
        1.  Invoke `setPendingAuthAction("save-meal")`.
        2.  Trigger an alert notifying the user they must sign in.
        3.  Redirect the user to the sign-in screen.
        4.  Abort execution.
3.  **Loading Guard:** If `isConvexAuthLoading`, abort execution.
4.  **Mutation Execution:**
    *   Set `isMutationLoading` to `true`.
    *   Wrap your `saveMeal` mutation call inside a `try/catch/finally` block.
    *   In `finally`, reset `isMutationLoading` to `false`.
